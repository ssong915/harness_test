# Harness Engineering 레슨런 — scenario1 케이스스터디

> 본 노트는 [[harness-engineering]]의 **갈래 B(Walking Labs 실행 런타임 하네스)** 를 `20-code/scenario1`에 직접 적용해 개발하면서 얻은 회고다. 이론(5개 서브시스템)이 "이래야 한다"고 말하는 지점에서 실제로 무엇이 샜는지를 대비한다.
> **위치 메모:** 회고/계획 성격이라 `99-meta/planning/`에 둠(불변 개념인 `00-concepts/` layer rule 회피, 기존 `harness-phase1.md`와 동계열). 내용이 안정화되면 `00-concepts/`로 승격 가능.
> **용어 브리지:** 본 노트의 **갈래 A = Part 1(의도)**, **갈래 B = Part 2(실행)** — [[harness-engineering]]가 2026-05-21 재구성되며 명칭이 바뀌었다. 아래 `§` 참조는 재구성된 절 번호 기준(§1 왜 하네스 / §2 파이프라인 / §3 Part 1 / §4 Part 2 / §5 9개 면 / §6 레슨런).

## 한 줄 요약
scenario1을 하네스 엔지니어링으로 개발하며, 이론상 하네스가 메워줘야 할 **의존성 가시성·컨텍스트 분배·세션 라이프사이클·검증 기준·병렬 실행**이 결국 사람(주원)의 수동 개입으로 메워졌다. 기대했던 "기획만 닫고 사람은 평가만" 모델이 작동하지 않았고, 이를 **스킬로 굳히는 것**이 다음 과제다.
> 🔴 **가장 치명적(L9):** "HERMES Agent"라는 **같은 단어**를 사람은 *실존 에이전트 제품*으로, 모델은 *이름만 같은 커스텀 LLM 에이전트*로 해석했다. 갈래 B(실행 하네스)는 멀쩡히 굴렀지만 **갈래 A(의도 닫기)가 비어 있어 "정확히 잘못된 것을 신뢰성 있게" 만들었다.** 시연 직전에 사람의 도메인 경험 + 사전에 말아둔 실제 HERMES 컨테이너로 겨우 복구.

## 0. 맥락

- **대상:** `20-code/scenario1` (HERMES 사고대응 시연, Claude API + Docker, 3인 분담)
- **방법:** Claude Code 기반으로 md/json 하네스(Agent.md, progress.md, handoff.md, feature-list.json, bootstrap-contract.md, init.sh, clean-state-checklist.md, reliability.md, architecture.md, sub-module.md …)를 구성해 개발.
- **1차 목표:** `feature-list.json`으로 기능 간 의존성을 파악하고 **병렬로 동시 개발**.
- **기대 운영 모델:** 초기 기획만 상세히 닫으면, 개발이 진행되며 검증·기록이 자동으로 채워지고, 사람은 마지막에 "잘 됐나" 평가만 한다.

## 1. 레슨 (주원 원본, ID 부여)

각 레슨은 독립 항목 — 다음 라운드에서 끼워넣기 쉽도록 ID로 관리.

### L1. 의존성이 명시적이지 않다
Python 코드는 어떤 라이브러리에 종속되는지, 파일들 간 의존이 어떤지 한 번에 분석된다. 그런데 md 등의 **에이전트 메모리·하네스 규격은 의존 관계가 명시적이지 않아** 아쉬웠다.
> **원하는 형태:** Obsidian 링크처럼 의존성을 표현할 수 있거나, **의존성을 관리하는 파일이 별도로 명시적으로** 존재.

**상세 (2차 인터뷰):**
- **의존성의 종류 = 셋 다였다** — ⓐ 동기화(A를 고치면 B도), ⓑ 읽는 순서, ⓒ 내용 참조(이 값의 진실은 저 파일). 그중 **핵심은 "읽는 순서"**: 순서를 명확히 해주면 됐지만, **"하나만 읽어"라고 하면 전체를 다 보지 못했다.** 즉 단일 진입점(Agent.md)에서 필요한 파일로의 **팬아웃(fan-out)이 신뢰성 있게 전개되지 않았다.**
- **실제로 난 사고 3종:**
  1. 의존성이 안 보여 **읽어야 할 파일을 못 읽음**.
  2. **중복 내용**을 여러 파일에 같이 적어둠.
  3. 작업이 완료돼 **업데이트됐어야 할 기존 내용이 잔존** → 나중에 **충돌과 환각**을 유발.
> → L1은 단순 "의존성 표시" 문제가 아니라 **(읽기 순서·팬아웃) + (잔존 상태 정리)** 두 축. 후자는 L3(종료·기록)·State와 직결.

### L2. 컨텍스트 분배가 매번 수동이다
`Agent.md`를 진입점으로 읽고, 거기서 `progress.md` / `handoff.md` / `feature-list.json` / `bootstrap-contract.md` / `init.sh` / `clean-state-checklist.md` / `reliability.md` / `architecture.md` / `sub-module.md` 등으로 **매번 손수 분배**해야 했다. 이 과정이 반복적이고, 신경 쓰기 어려웠다.

**상세 (2차 인터뷰):**
- **분배 주체 = 사람.** 세션 진입 시 주원님이 "Agent.md를 읽어" 식으로 **읽어야 할 파일을 직접 지정**했다. 파일 개수가 **100개를 안 넘어** 그 방식으로 커버는 됐다(=규모로 버틴 것).
- **새 리스크 — 브랜치 오염:** 작업 브랜치가 설정 안 된 상태로 진입하면 **갑자기 다른 사람의 브랜치 작업을 세션에서 진행**해버리는 사고가 났다.
- **취약점:** 명시적으로 알려주면 괜찮았지만, **지정이 누락되면 그대로 준수 실패** — 사람의 기억에 의존하는 구조라 깨지기 쉬웠다.
> → L2의 본질은 "수동이라 번거롭다"를 넘어 **사람 지정에 의존 → 누락 시 무방비**. 특히 브랜치 가드는 Lifecycle/Scope의 안전장치로 자동화 1순위.

### L3. 세션 종료·기록을 매번 챙겨야 한다
기록도 세션이 끝날 때마다 해야 하고, **세션 종료 작업이 잘 돼야** 다음이 이어지는데, 매번 신경 쓰기가 어려웠다.

**상세 (2차 인터뷰) — 세션 종료 작업의 실제 항목:**
1. 전역 `Agent.md` 등 md가 **잘 준수되고 있는지** 점검.
2. **수정돼야 할 내용**이 있었는지 점검.
3. 내가 진행한 **모듈/브랜치의 md·json이 갱신됐는지**, 그리고 (일반 코드 파일 외에) **작업 항목이 메모리 파일에 기록됐는지 / 제대로 분배돼 기록됐는지** 확인.
4. 위를 **보고하고 남기는** 과정.
5. 추가로 `init.sh`, **새로 시작 전 만들어야 하는 체크리스트** 확인.
> → 종료 작업은 사실상 **"잔존 상태 정리(L1-사고3) + 기록 분배(L2) 검수"** 의 합이다. L1·L2·L3가 같은 라이프사이클 루프의 앞/뒤라는 게 드러남.

### L4. 테스트→수정 루프가 번거롭다
e2e로 검증한 결과를 (사람이) 다시 알려주고, **수정할 때도 분배를 다시 지시**해야 하는 게 번거로웠다.

**상세 (2차 인터뷰):**
- **e2e 검증 = 사람 + 에이전트 협업.** 에이전트는 **웹소켓 통신 등**을 바라보고, 사람은 **웹페이지·휴대폰에서 실제 동작을 관측**해 안 되는 항목을 찾아 알려주는 방식.
- **"수정 시 다시 분배"의 정체:** 안 되는 항목을 고치려면 *무엇을 해야 하는지*를 다시 명확히 해서 md·json에 **분배**하는 작업. **이 분배가 끝나야 다음 세션이 별도 프롬프트 지시 없이 작업 가능**하기 때문에 생략할 수 없었다.
> → L4는 L2(분배)·L3(기록)와 같은 메커니즘의 *수정 사이클* 버전. "관측→원인규명→분배"가 매 수정마다 사람 손을 거쳐야 했다.

### L5. 병렬 작업이 직렬로 무너졌다
1차 목표는 `feature-list.json`으로 의존성을 파악해 **여러 세션에서 병렬로 동시 작업**하는 것이었다. 그런데 막상 하니 명확히 분리되지 않았는지 **한 세션에서 계속 작업**하는 상황이 됐다. Claude Code를 써서 **worktree가 분리되고 파일도 실제로 다른 공간에서 작업되는데도** 이 문제가 발생했다. 멀티 에이전트 뷰로 한 번에 여러 세션을 굴리고 싶었는데 이 부분이 아쉬웠다.

**상세 (2차 인터뷰) — 직렬화의 근본 원인 (복합):**
1. **메타 파일 충돌이 가장 컸다.** 기능(코드)은 분리됐지만 **메타 파일(progress/handoff/feature-list 등)이 공유**돼서 세션들이 같은 파일을 건드림.
2. **의존성 이슈**도 있었다(L1).
3. **워크트리/브랜치 분리 실패** — git 브랜치는 나눴지만 **워크트리가 분리 안 됐거나 브랜치가 같아서** 충돌.
4. **운영 부담** — 전체 컨텍스트를 사람이 다 기억하며 지시해야 했고, 세션마다 "병렬 가능한가 / 충돌 나는가"를 매번 고민해야 했다.

**상세 — 병렬을 막은 *대기·권한* 병목:**
- **검증 대기:** 하네스가 한참 걸려 작업이 다 되면 그제야 확인·입력 → 왕복이 오래.
- **파이프라인 정지:** 검증 중 다른 git 브랜치 충돌이 안 끝나면 기다리며 파이프라인이 멈춤.
- 🔑 **권한(permission) 병목:** git 명령이 `deny`라 사람에게 요청 → 느림. `ask`로 바꿨더니 그것도 **중지된 채 대기** + **사람의 판단을 계속 가져가서** 문제. (혹시 잘못될까 봐 권한을 좁게 둔 것.)

> 💡 **주원의 해법(설계 원칙):** *항상* 개발 기능별로 **브랜치를 분리하고 세션도 분리**했다가, 머지하며 **브랜치를 삭제**한다. 그러면 **워크트리도 자연히 분리**되고, "git이 꼬일까" 걱정이 사라지므로 **권한을 넉넉히 줘도 안심** → `deny/ask` 병목과 메타 파일 충돌이 함께 풀린다.

**멀티 에이전트 뷰 (Q8):** 멀티 에이전트 뷰의 문제는 **위 멀티 세션 문제와 동일**하다. 별도의 새 문제가 아니라, 메타 파일 충돌·워크트리 미분리·권한 병목이 동시 세션에서 그대로 터진 것.

> ✅ **진단 완료** — 직렬화 = (메타 파일 공유) + (워크트리/브랜치 미분리) + (운영 부담) + (대기·권한 병목)의 복합. → §6 Q7/Q8 해소.

### L6. "무엇을 검증해야 하는지"가 없다
테스트 자체와 **pre-commit 훅으로 테스트를 거는 부분은 좋았다**. 다만 e2e로 검증할 때 매번 확인하기 불편했고, **기능이 개발될 때마다 "무엇을 검증해야 하는지"를 알 수 있었으면** 더 좋았겠다.

**상세 (2차 인터뷰) — 검증 기준의 위치 (Q9):**
- 검증 기준은 **두 층위**: ⓐ **전체(시스템)** 에 대한 것, ⓑ **기능**에 대한 것.
- 기능 단위는 **유닛테스트 등으로 처리**되니, 기능들이 구현돼 묶이는 **모듈 단위**가 보통 있다 → **그 모듈에 대해 한 번에 검증**할 수 있게 한다.
- 효과: **매 세션 검증할 필요가 없고**, *모듈 완성 타이밍*마다 "이제 완성됐구나, 검증할까"라는 자연스러운 흐름이 생긴다.
- **위치:** `feature-list.json`은 "기능들의 내용·목록"을 기록하는 곳이라 검증 기준을 넣기엔 안 맞는다. → **별도 파일**로 분리하는 게 맞다.
> → 내 v0.3 제안(feature-list 인라인)을 교정. 검증 기준은 **모듈 단위로 묶인 별도 검증 파일**이 옳다(D3 수정).

### L7. 세션 라이프사이클이 자동으로 닫히지 않는다
기능 하나를 개발하려면 — 세션 시작 시 필요한 md를 다 확인하고, 커밋·브랜치를 확인하고, 작업 하나를 진행하고, **세션 종료 작업으로 기록을 역으로 다 남기고** 끝내야 한다. 이게 **별도로 시키지 않아도 알아서 되길** 바랐는데 잘 안 됐다.

### L8. 기대한 운영 모델 vs 실제
기대: ① **초기 기획만 상세히** 닫는다 → ② 개발하며 **e2e 검증**을 진행한다 → ③ 그 과정에서 **알아서 md에 기록**된다 → ④ 사람은 최종적으로 **"잘 됐나" 평가만** 한다. 이렇게 기능이 개발되며 스스로 해결돼 가는 흐름을 기대했으나 아쉬웠다.
> **결론:** 신경을 덜 쓰도록 **스킬로 존재**하면 좋겠다.

**상세 (2차 인터뷰) — 누수 빈도·기획 상세도 (Q10):**
- **누수 빈도 = "사람이 깜박하고 입력 안 하면" 샜다.** 상시가 아니라 사람 트리거 누락 시.
- **부분 자가교정이 이미 일어났다:** 그걸 확인한 뒤 **md들에 세션 시작/종료 + 하네스 엔지니어링 필요 내용을 잘 정돈해 넣으니**, 이후엔 **"Agent.md 읽고 준비해"라는 말만으로도 충분**해졌다.
- **초기 기획이 *엄청* 부족했던 건 아니다 — 다만 더 *날카롭게* 다듬을 여지는 있었다.** 세션 라이프사이클·검증·기록 규약을 처음부터 좀 더 촘촘히 박았으면. (※ "촘촘한 기획"의 정의 = 충분히 길게 쓰는 것이 아니라, **필요한 지점에서 상세히 묻고 닫고 진행**하는 것 — 주원의 "잘 된 기획" 감각.)
> → 시사점: 사람 기억에 의존하는 트리거는 *반드시* 샌다. 진입점(Agent.md)에 규약을 잘 정돈하면 *지시 비용*은 크게 줄지만(이미 달성), **트리거 자체의 자동화**(훅)는 여전히 D2의 몫.

### L9. 🔴 의도 오해 — "이름만 같은 가짜 HERMES"를 신뢰성 있게 만들어버림 *(가장 치명적)*
원했던 것은 **HERMES Agent**, 즉 OpenClaw·NemoClaw 같은 **에이전트로서의 실존 HERMES Agent**(실제 있는 서비스/제품)였다. 그런데 개발된 것은 **"HERMES Agent"라는 이름의 일반 LLM**으로 에이전트 기능을 하는 에이전트였다. 기능적으로는 유사했지만 **실제 있는 서비스를 쓴 게 아니라 이름만 같게 커스텀 HERMES를 만들어버린** 것. 그래서 시연 전에 시간도 없이 **전체 구조를 힘들게 갈아엎어야** 했다.

**무엇이 구해줬나 (복구 자산):**
- **사람의 폭넓은 도메인 경험** — 인프라·보안·웹·웹소켓·미디어·NPU 설계·AI 최적화·프롬프트 엔지니어링·Efficient Inference for LLM 등 직접 해본 경험에서 **빠르게 방향을 제시**.
- **사전에 말아둔 실제 HERMES** — 이미 도커 이미지로 HERMES를 말아 컨테이너로 쓰던 게 있어, 그것을 제공하고 방안을 제시하며 해결.

> **본질:** [[harness-engineering]] §3의 *same word, different ontology*. "HERMES Agent"라는 한 단어가 사람·모델에게 다른 영역을 가리켰다. 갈래 B(실행 하네스)는 "이름이 HERMES인 에이전트가 동작한다"는 통과시켰지만, **"그게 실제 HERMES 서비스인가(=*what*)"는 아무도 묻지 않았다.** L6(검증 기준)과 직결 — acceptance criteria에 "실존 HERMES 서비스를 사용한다"가 있었다면 검증에서 걸렸을 것.
> **역설:** 복구를 가능케 한 것이 결국 **사람의 도메인 깊이**였다는 점 — 감시 비용이 0으로 가긴커녕 위기 순간에 사람에게 폭증했다. 하네스가 의도를 못 닫으면 *가장 비싼 순간*에 사람이 소환된다.

**상세 (2차 인터뷰):**
- **발견 시점 = 시연 *전날* (Q11).** **스킬을 직접 주입하려는데 안 돼서** 비로소 드러남. 그 전까지 나머지 기능은 다 됐고, **물어보면 "나는 HERMES"라고 답**해서 진짜인 줄 알았다. → *가짜가 자기 정체를 긍정하니 통과*해버린 것.
- **근본 원인 (Q12):**
  - **정체성 확인 절차 부재** — 명칭이 명확("HERMES Agent")했기에 "혹시 아니라면 확인"하는 게이트가 있어야 했는데 없었다.
  - **구조를 그대로 따라가서** "맞다"고 느낌.
  - 🔴 **풍부한 배경 컨텍스트가 오해를 *강화*함** — 이전에 `00-concepts` 등 여러 곳에 HERMES agent 조사가 많아, 그 컨텍스트가 "이게 진짜 HERMES"라는 확신을 키웠다. → **"Context는 Contract가 아니다"의 완벽한 실증**: 배경 문서가 많을수록 모델은 *더 자신 있게 그럴듯한 동명 가짜*를 만든다.
- **복구 비용 (Q13):** 예상 **6일** → 미리 말아둔 **도커 컨테이너를 활용하도록 가이드**해 **몇 시간** 만에 갈아엎음. 재현은 가능하나 **복잡함**.
> → **D5 강화 근거:** 그 도커 자산이 *하네스에 등록*돼 있었다면 애초에 가짜를 안 만들었을 수 있다. 자산 레지스트리 + **정체성 확인 게이트**(AC: "물어봤을 때 이름이 맞다고 답하는 것만으론 불충분 — 실존 서비스 연결 증거 요구")가 필요.

### L10. 관측성 부재 — 세션 용량·하네스 효과를 측정 못 함
*(B2·B3 답변에서 도출된 신규 차원)*
- **세션 용량 판단 불가 (B2):** 어떤 모델이 단일 세션에서 *어디까지* 효율적으로 잘 되는지 판단하기 어려워, **휴리스틱하게 컨텍스트량을 보면서** 진행했다. "지금 세션을 끊어야 하나"를 감으로 정함.
- **하네스 효과 측정 불가 (B3):** `reliability.md`·`architecture.md`가 **실제로 도움이 됐는지 인지하지 못했다.** 백그라운드에서 동작한 건지, 아예 안 쓰인 건지 알 수 없음 → *효과를 측정할 수 없는 하네스 산출물*은 [[harness-engineering]]의 실패 모드 "기준 과잉"에 해당할 위험.
> → 둘 다 **관측성(observability)** 결함. 하네스가 "지금 컨텍스트 예산이 얼마 남았고 / 어느 산출물이 실제로 참조됐는가"를 보여주지 못했다. ([[harness-engineering]] "왜 observability가 하네스 안에 속해야 하는가"와 정확히 일치.) → D7.

- **L2·L3·L4·L7은 한 부류:** "매 세션 반드시 반복돼야 하는데 사람이 잊기 쉬운 의식(ritual)". → 자동화/훅의 대상.
- **L6은 Verification ↔ Scope의 연결고리:** `feature-list.json`의 "done 정의"에 검증 기준이 함께 있어야 한다는 신호.
- **L9는 갈래 B 프레임 전체의 사각지대다.** scenario1은 갈래 B(실행 런타임 하네스)에 집중했지만, 가장 치명적인 실패는 **갈래 A(의도 닫기)의 부재**에서 왔다 → §2.5.
- **권한(permission) 모델도 5-서브시스템에 없는 신규 차원이다(L5).** deny→사람요청(느림) / ask→중지·판단 탈취. 좁은 권한이 *안전*을 사지만 *병렬성·자율성*을 죽인다. 주원의 해법(브랜치·세션·워크트리 격리 → 권한 확대)은 **"격리로 안전을 확보해 권한을 풀어주는"** 교환 — D6.

### 2.5 L9가 드러낸 것 — 갈래 A가 통째로 비어 있었다

5개 서브시스템(갈래 B)은 **"running agent를 안 망가뜨리는 법"** — *언제·어디서·어떻게(when/where/how)* 를 통제한다. 그러나 **무엇을(*what*) 만들 것인가**는 갈래 A(Seed·AC·ontology)의 몫이다([[harness-engineering]] §2 표).

scenario1은 갈래 B는 둘렀지만 갈래 A — "HERMES Agent의 *의미*를 실행 전에 계약으로 닫기" — 가 없었다. 그 결과:
- 검증(L6)은 *기능 유사성*만 봤고 *정체성*("실존 HERMES인가")은 안 봤다.
- "Context는 Contract가 아니다"([[harness-engineering]] §1)의 교과서적 실패 — HERMES 배경 문서를 아무리 줘도, **검증 가능한 계약**으로 "실존 서비스 사용"을 박지 않으면 모델 해석에 맡겨진다.
- → **이번 회고의 가장 큰 메타 교훈:** 실행 하네스(갈래 B)를 스킬화하기 *전에*, 의도 하네스(갈래 A) — Seed Gate로 *what*을 닫는 단계 — 를 파이프라인 앞단에 반드시 둬야 한다. ([[BigHarness]]·[[Symposium]] 결합 그림과 동일.)

## 3. 기대 운영 모델 vs 실제 (L7·L8 확대)

| 단계 | 기대 (자율) | 실제 (수동) |
|---|---|---|
| ① 기획 | 초기에 상세히 닫음 | **엄청 부족했던 건 아니나, 더 *날카롭게* 다듬을 여지** — 라이프사이클·검증·기록 규약을 더 촘촘히 박았으면 |

→ 기대의 핵심은 **"감시 비용(supervision cost) 최소화"** ([[harness-engineering]] §1). scenario1에선 ②③이 사람에게 되돌아오며 감시 비용이 줄지 않았다.

## 4. 개선 아이디어 / 스킬화 방향 *(stub — 추후 확장)*

> 🎯 **이 회고가 향하는 산출물(목표):** 레슨이 글로 끝나지 않고 **재사용 가능한 하네스 자산**으로 굳는 것 —
> ① 전체 내용을 담은 **종합 md 문서**(= 이 노트), ② 그 내용을 반영한 **`Agent.md`류 파일들**, ③ 실제로 바로 쓰는 **템플릿**, ④ 이를 자동 적용하는 **스킬**.
> 아래 D1~D7은 그 자산에 들어갈 **설계 요소**들이다. ("충분한 기획" = 길게 쓰기가 아니라 *필요한 지점에서 상세히 묻고 닫고 진행*하는 것.)

> 아직 정제 전. 레슨에서 자연스럽게 파생된 후보만 먼저 적어둔다.

- **D1. 의존성 매니페스트 + 읽기 순서 (L1):** 하네스 파일·기능 간 의존을 `deps`로 명시(Obsidian 링크식 또는 별도 `dependencies.json`). 단순 의존 표시를 넘어 **(a) 진입점에서의 읽기 순서/팬아웃을 기계가 전개**("하나만 읽어"가 아니라 그래프를 따라 필요한 전부를 끌어옴), **(b) "이 파일 고치면 같이 갱신할 파일"**을 알게, **(c) 작업 완료 시 잔존 내용 정리 트리거**(중복·옛 상태 → 충돌·환각 방지).
- **D2. 분배·라이프사이클 스킬 (L2·L3·L7):** 세션 시작(의존성 그래프로 필요 md 자동 선별 로드 + **브랜치 가드** — 지정 누락/타인 브랜치 진입 차단) → 작업 → 종료(§1 L3의 5개 체크 항목을 역방향 자동 수행·clean-state)를 **스킬 + 훅**으로. 사람 기억 의존 제거 — 누락돼도 무방비가 되지 않게.
- **D3. 모듈 단위 검증 파일 (L4·L6) — *v0.5 수정*:** ~~feature-list.json 인라인~~이 아니라, **별도 검증 파일**(예: `verification.json` / `acceptance/`)에 **(ⓐ 전체 시스템 / ⓑ 모듈 단위)** 검증 기준을 둔다. 기능 단위는 유닛테스트로, **모듈은 완성 시 한 번에 e2e 검증**. 모듈 완성 타이밍이 곧 검증 트리거 → "매 세션 검증" 부담 제거. feature-list.json은 기능 목록 본연의 역할만 유지(관심사 분리).
- **D4. 병렬 오케스트레이션 — 메타 파일 격리 (L5):** 직렬화의 1번 원인이 **메타 파일 공유**로 확인됨. progress/handoff/feature-list를 **worktree·세션별로 분리**(또는 충돌 없는 append/merge 규약)해 동시 세션이 같은 파일을 안 건드리게. 멀티 에이전트 뷰 동시 작업의 전제조건.
- **D5. 의도 하네스 + 자산 레지스트리 (L9) — 최우선:** 실행(D1~D4) *앞단*에 **갈래 A(의도 닫기)** 를 둔다. 핵심은 **"의도가 모르게 커스텀 구현으로 변질되는 것(intent drift)"을 막는 것** — L9의 가짜 HERMES는 그 *한 사례*일 뿐이다. ⓐ 핵심 명사·산출물의 *의도*를 실행 전에 한 줄로 닫는 Seed Gate(→ [[Symposium]]식 `/socrates`), ⓑ acceptance criteria에 **"의도한 실물/자산을 쓴다(임의 동명 커스텀 금지)"**를 명시(필요 시 정체성 확인까지), ⓒ 이미 보유한 자산(예: 도커로 말아둔 실제 HERMES 컨테이너)을 하네스가 인지하는 **자산 레지스트리**(`assets.json` 등)로 "새로 만들지 말고 이걸 써라"를 못박기. *주의: 배경 컨텍스트가 많을수록 변질된 구현을 더 그럴듯하게 만드므로(Q12), Context 주입이 아니라 검증 가능한 Contract로 닫아야.*
- **D6. 브랜치 격리 기반 권한 확대 (L5) — 핵심 설계 원칙:** 모든 작업을 **기능별 브랜치 + 분리된 세션/워크트리**에서 → 머지 후 **브랜치 자동 삭제**. 격리로 "git 꼬임" 위험이 사라지므로 **권한(특히 git 명령)을 넉넉히 부여**할 수 있다 → `deny`(사람 요청 대기)·`ask`(중지·판단 탈취) 병목 해소 + 메타 파일 충돌 동시 해결. *L5의 모든 하위 원인(메타 충돌·워크트리 미분리·권한·대기)을 한 번에 겨냥하는 단일 레버.*
- **D7. 관측성 — 컨텍스트 예산 + 산출물 사용 추적 (L10):** ⓐ 세션의 **남은 컨텍스트 예산/포화도**를 표면화해 "언제 세션을 끊을지"를 감이 아니라 신호로(→ [[bh-monitor]]류 대시보드 연계 가능), ⓑ 어떤 하네스 산출물(reliability/architecture 등)이 **실제로 참조·기여했는지** 로깅 → 효과 없는 산출물은 정리(기준 과잉 방지).

## 5. 워크스페이스 좌표

- 이 회고는 [[BigHarness]] / AlphaClaw 철학과 같은 방향이다 — "기획 닫고 → 자율 실행·기록·검증 → 사람은 평가만"은 BigHarness가 SWE-bench로 달성하려는 것과 동일. **scenario1은 그걸 손으로 돌린 셈**이고, 그래서 "스킬로 굳히고 싶다"가 자연스럽다.
- D2~D3의 스킬화는 [[harness-engineering]] §4의 State·Verification·Lifecycle을 scenario1 맥락에서 구현하는 일.
