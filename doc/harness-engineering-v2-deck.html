Shared MeaningPremise

같은 단어를 써도,
서로 다른 영역을 본다.

"성과"라는 한 단어 안에 매출, 학습 속도, 팀 분위기, 출시 속도, 채용 성공률이 다 들어 있다. 단어 자체는 이 영역을 통째로 가리킨다.

그래서 한 명은 매출을, 모델은 학습 속도와 팀 분위기를 떠올린다. 같은 단어를 쓰지만 본 것이 다르다 — 협업의 어긋남은 거의 여기서 시작된다.

하네스의 앞단은 그 단어가 어디까지 가리키고 어디부터는 아닌지를 함께 닫는 일이다.

Part 2를 단단히 하는 교훈

State

상태 소스는 하나 — dev-features를 폐기하고 모듈별 feature-list 하나만 단일 진실. 진입에서 필요한 전부로 팬아웃을 닫아라.

Verification

완료 판정은 사람만, 검증은 실 LLM 작동으로. mockup·스크립트 재생이 아니다 — "실제 응답을 처리 못 한다"가 가치.

Scope / 병렬

병렬은 메타 파일 공유로 쉽게 직렬로 무너진다. 브랜치 격리 → 권한 확대가 해법.

Lifecycle

핸드오프는 정규 파일에만, NEXT_SESSION 같은 임시 파일 금지. 세션은 clean state로 끝낸다.

Observability

"옛 동작 보임" → 코드 가설 전에 ① 서버 재기동 ② 정적 JS 캐시부터. 라이브 reproduce가 가설보다 빠르다.

Observability ②

세션 컨텍스트 예산·산출물 사용이 안 보인다 — 표면화해야 할 미해결 과제.
